🏢 Scenario

The Brownfield Challenge

Adding a new feature to an existing .NET 8 billing service — a codebase that has been in production for 3 years with multiple developers having contributed to it.

🏢

Why brownfield is different from greenfield: AIDLC cannot assume a clean slate. Before generating any requirements or code, AI must first understand the existing architecture, patterns, and constraints — otherwise it will generate code that conflicts with existing conventions, creates duplicate abstractions, or misses integration points. The Reverse Engineering stage is mandatory for brownfield projects.

The Business Requirement

📋 Azure DevOps — Work Item #5234
STORY #5234 · PO: Sarah T. · Blocking: Enterprise sales · Priority: Critical
Bulk invoice export — CSV and PDF
Enterprise customers need to export invoice history to their accounting systems. They currently export one at a time (2–3 hours of manual work for 100+ invoices/month).

Request: Add a bulk export endpoint to the existing billing-service. Users filter by date range and status, then download as ZIP (individual PDFs) or single CSV.

Constraints: Max 500 invoices. Finance and Admin roles only. Must not degrade existing invoice viewing performance. Asynchronous for large exports.

The Existing Codebase (from Azure DevOps)

billing-service — repo stats
Language: C# (.NET 8)
Age: 3 years, 24 contributors
Lines of code: ~38,000
Solution projects: 6 (API, Core, Infrastructure, Background, Tests.Unit, Tests.Integration)
Known patterns: CQRS (MediatR), Repository, Azure AD, Serilog, xUnit
⚠ Partial migration: some old-style controllers alongside CQRS handlers
⚠ Two different PDF generation libraries used across old features (iTextSharp + PdfSharp)
🔴 Brownfield Risks
  • AI may generate code that clashes with partial migration state (old controllers vs CQRS)
  • Two PDF libraries already present — AI must choose one, not introduce a third
  • Existing endpoint conventions may differ from ideal patterns — AI must match them
  • Performance risk: export query must not block existing invoice views on shared SQL instance

📡 Inception — Requirements Analysis

Context-Aware Requirements Questions

Because AI has analysed the codebase, its clarifying questions are specific — not generic. It already knows the data model, auth patterns, and existing export conventions.

🤖 GitHub Copilot — Inception Phase (codebase-aware)
GitHub Copilot — Context-Aware Questions
Based on my codebase analysis, I have specific questions (not generic ones):

1. Export pattern alignment: I found an existing async job pattern at StatementExport: POST to start → poll status → download. Should the invoice bulk export follow this same pattern, or is synchronous acceptable for small exports (<50 invoices)?

2. PDF source: The Invoice entity already has a PdfStoragePath field pointing to Azure Blob. For the ZIP export, should I retrieve the pre-generated PDFs from Blob Storage, or re-generate PDFs from the database data?

3. Filter scope: The existing GetPagedByFilter repository method supports CustomerId + date range + Status. Is this sufficient for the export filter, or do you need additional filter fields (e.g. minimum amount, specific invoice IDs)?

4. Role scope: I see Finance and Admin roles in your auth patterns. The brief says "Finance and Admin only" — confirmed correct, no new roles needed?
Sarah (PO) + Jamie
1. Follow the existing async job pattern for all exports — consistency is more important than latency.
2. Use pre-generated PDFs from Blob Storage if they exist (99% of cases). Fallback: re-generate from DB if PdfStoragePath is null (rare edge case for old invoices).
3. Existing filter fields are sufficient for MVP.
4. Finance and Admin confirmed — no new roles.
GitHub Copilot
✅ All questions answered. One additional finding from the schema analysis:

⚠️ Scale risk: A max-500-invoice export generating PDFs from Blob Storage will make up to 500 concurrent Blob reads. I recommend batching reads (50 at a time) to avoid Azure Storage throttling. I'll include this as a technical constraint in the requirements.
💡

Compare to greenfield: In a greenfield project, AI would ask "what is your authentication model?" and "what's your data schema?". In this brownfield session, AI already knows the answers from Reverse Engineering — it only asks questions about decisions that can't be inferred from code.
