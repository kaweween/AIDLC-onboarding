General Questions

All Roles
Q What exactly is AIDLC?

AIDLC (AI-Driven Development Lifecycle) is a software development methodology where AI is positioned as a central collaborator — not a passive tool — throughout the entire development process. It was formalised by AWS in July 2025, open-sourced in November 2025, and is now being adopted by engineering teams globally regardless of their cloud provider.

The methodology has three phases: Inception (requirements and design), Construction (code and testing), and Operations (infrastructure and deployment). At each stage, AI creates plans, asks clarifying questions, and implements only after explicit human approval. No phase transition happens without a human sign-off checkpoint.

Q Is this just a rebranding of Agile/Scrum? What's actually different?

No — AIDLC is a genuine structural change, not a rebrand. The differences are substantive:

  • Who creates artefacts: In Agile/Scrum, humans write requirements, stories, and code. In AIDLC, AI drafts all artefacts; humans validate and approve them.
  • Cycle length: Sprints are 2 weeks. AIDLC "Bolts" are hours to days. The rhythm is calibrated to AI's speed, not human capacity.
  • Collaboration model: SDLC work is largely asynchronous and individual. AIDLC replaces this with synchronous "Mob" sessions where the whole team engages AI together in real-time.
  • Documentation: In SDLC, documentation is often an afterthought. In AIDLC, every artefact (requirements, design, audit trail) is auto-generated and committed to the repository as the work progresses.
Key insight
Agile was a response to waterfall's slowness. AIDLC is a response to Agile's assumption that humans must do most of the work themselves.
Q Won't AI just generate wrong or hallucinated requirements and code?

This is the most common concern, and AIDLC's core design directly addresses it. The methodology's answer is: AI always asks before acting, and humans always approve before AI proceeds. Specifically:

  • AI presents requirements as questions, not assertions. The team corrects errors in the Mob Elaboration session before anything is committed.
  • Mandatory checkpoint gates prevent AI from proceeding to Construction until Inception artefacts are human-approved.
  • Every AI-generated artefact is stored and versioned. A hallucinated requirement will be visible in the review, not buried in code.
  • For code, the key rule is: developers must understand and be able to explain every line of AI-generated code before it merges. AI generates a first draft; engineers review, challenge, and modify it. The bar for merging is the same as for human-written code.
Q What are the biggest risks of adopting AIDLC?
  • Over-reliance on AI output: Teams may reduce scrutiny of AI-generated code over time. Requires ongoing governance and code review discipline.
  • Context quality: AIDLC is only as good as the context you give it. Poorly written steering rules or vague problem statements produce poor AI output. Requires up-front investment in quality context-building.
  • Human approval bottlenecks: If approvals are slow, the productivity gains evaporate. AIDLC requires humans to be more available, not less.
  • Security of AI-generated code: AI may reproduce patterns from training data that include known vulnerabilities. All AI-generated code must pass the same security scanning (SAST/DAST) as human-written code. Do not reduce security gates.
  • Skill atrophy: If developers stop writing code from scratch, junior engineers may not develop foundational skills. Teams should maintain code-writing practice alongside AIDLC adoption.
  • Perceived productivity vs actual productivity: Early research found a gap between how productive teams felt using AI and their actual measured output. Measure outcomes (shipped features, defect rates) not activity (lines generated, prompts sent).
Q Do we need to change our team structure?

No immediate team restructuring is required. Roles evolve within the existing structure:

  • Developers shift from writing code to reviewing and guiding AI-generated code
  • POs shift from writing requirements to validating AI-generated requirements
  • Architects shift from designing systems to approving AI-proposed designs and tuning steering rules
  • SMs shift from running ceremonies to facilitating Mob sessions

Over time, teams may find they need fewer people doing routine implementation work and more people doing strategic technical decision-making — but that is a gradual evolution, not a day-one restructure.

Using AIDLC Without AWS

All Roles
Q AIDLC was created by AWS. We don't use AWS. Does this apply to us?

Yes — completely. The AWS team open-sourced the AIDLC workflow rules under the MIT-0 licence in November 2025. The methodology itself is tool-agnostic and cloud-agnostic. Here's how it maps to our stack:

  • The core AIDLC workflow is implemented as a set of text files (rule files) placed in your repository. Any AI coding agent that reads project-level instructions can run AIDLC.
  • GitHub Copilot (which integrates directly with Visual Studio and Azure DevOps) supports AIDLC via the .github/copilot-instructions.md file — no AWS tools required.
  • Cursor supports AIDLC via .cursor/rules/ directory.
  • Claude supports AIDLC via CLAUDE.md at project root.
  • AWS-specific tools (Amazon Q Developer, Kiro) are one implementation of AIDLC — not the only one.
For Our Stack
We use GitHub Copilot within Visual Studio, which natively supports AIDLC. Our Azure DevOps repos and Jenkins pipelines require no changes. The methodology sits on top of our existing infrastructure.

Q Is there a risk that AWS will make AIDLC proprietary or change the licence?

The AIDLC workflow rules are released under the MIT-0 licence, which is one of the most permissive open-source licences available. It has no attribution requirement and places essentially no restrictions on use, modification, or redistribution.

The methodology itself (documented in the whitepaper and AWS blog) is published as an open standard. Other vendors (Microsoft/GitHub, Cursor, Anthropic) have already incorporated or acknowledged it. Even if AWS were to stop developing it, the community and competing implementations would carry it forward.

Recommendation
Pin your AIDLC rule files to a specific release tag in your repo. Don't auto-update from the upstream repo without review. This insulates you from any upstream changes.

For Software Engineers

Engineers
Q Will AIDLC replace my job?

No — and the evidence from early adopters supports this. AIDLC changes what engineers do, not whether they're needed. The shift is from:

  • Writing boilerplate, glue code, and routine implementation → to: reviewing, guiding, and approving AI-generated code
  • Writing documentation and tests → to: validating AI-generated docs and tests for correctness and completeness
  • Implementing well-understood patterns → to: solving novel problems, system design, and architectural decisions that AI cannot make confidently

Teams that adopted AIDLC did not reduce headcount — they increased output per person. The industry is moving toward needing more engineers who understand AI systems well, not fewer engineers overall.

The Real Risk
Engineers who don't engage with AIDLC and continue to work in fully manual workflows may find their productivity gap vs AI-augmented peers widens significantly over the next 2–3 years.
Q How do I trigger AIDLC in our repos?
  1. Ensure the AIDLC steering files are in your repo (see setup guide in the Onboarding Kit)
  2. Open GitHub Copilot chat in Visual Studio
  3. Start with the trigger phrase: "Using AI-DLC, [your intent here]"
  4. Example: "Using AI-DLC, I want to add pagination to the customer list endpoint in our billing service"
  5. Follow the AIDLC workflow — it will ask clarifying questions, detect if the project is brownfield, and guide you through Workspace Detection, Requirements Analysis, and into Construction
Important
Only use the trigger phrase for new pieces of work that benefit from the full AIDLC workflow. For quick bug fixes or one-line changes, normal Copilot chat is fine. AIDLC is optimised for features and meaningful changes.
Q How much of the AI-generated code should I actually read and review?

All of it. This is a non-negotiable rule in AIDLC. AWS's guidance explicitly states: "developers must understand every line of AI-generated code." This is not optional, even if the code looks obviously correct at a glance.

Practically, this means:

  • Review AI-generated code with the same depth as you would a junior engineer's code
  • Run the code locally and trace through the logic for non-trivial functions
  • Be prepared to explain any line in a code review — if you can't explain it, it's not ready to merge
  • Pay extra attention to security-sensitive code (auth, data access, input handling), even if the AI was instructed on your security standards
Q What if the AI generates code that violates our coding standards or architecture patterns?

This is where the quality of your steering rules matters. AIDLC is designed so that coding standards, design patterns, security requirements, and architectural decisions are encoded in the steering files (rule files) that AI reads before generating anything. If AI repeatedly violates a standard, the fix is to update the steering rules, not to keep correcting it manually.

Short-term, simply reject or modify the AI's output in your review. Long-term, flag it and work with your architect to update the relevant rule file so the standard is consistently enforced.

For Software Architects

Architects
Q How does AIDLC handle architecture decisions? Does AI just pick a pattern?

During the Construction phase's Functional Design stage, AI proposes a logical architecture based on the context accumulated during Inception. It does not make autonomous architecture decisions — it presents options and asks for your approval before proceeding. The flow is:

  1. AI analyses the requirements and existing codebase context
  2. AI proposes an architecture with rationale (stored in aidlc-docs/construction/design.md)
  3. AI asks structured clarifying questions about constraints, preferences, and non-negotiables
  4. Architect reviews, modifies the proposal, and explicitly approves before code generation begins

Your role as architect shifts from designing in isolation to guiding and approving AI proposals in real-time during Mob Construction. You spend less time on documentation and more time on architectural judgment.

Q How do we encode our existing architectural standards into AIDLC?

Through the steering rule files. The AIDLC open-source rules give you a starting framework. You extend them with your organisation's specific standards by adding content to the rule files or creating additional context files. Key things to encode:

  • Preferred design patterns for your domain (e.g., "use the repository pattern for data access")
  • Technology constraints (e.g., "do not introduce new dependencies without architect approval")
  • Security requirements (e.g., "all endpoints must use Azure AD authentication")
  • Logging and observability standards
  • API design guidelines
  • Database conventions

These go in a "semantic context" document that is part of your repo's AIDLC rule set. It's a living document — update it when standards evolve.

Q How does AIDLC handle brownfield/legacy systems where the codebase is large and complex?

AIDLC has a dedicated Reverse Engineering stage for brownfield projects. Before any Inception work begins, AI analyses the existing codebase to build a semantic context map — understanding the existing architecture, patterns, and dependencies. This is critical and typically takes 15–60 minutes depending on codebase size.

The key challenge is "semantics-per-token ratio" — AI can only hold a certain amount of context at once. For large codebases, this means:

  • You should pre-prepare a concise architecture summary document that captures the key decisions, not just the code
  • Scope work items to modules or bounded contexts rather than the full system
  • Update the semantic context document as the codebase evolves
Legacy Risk
AI trained on public code may suggest patterns that don't match your internal conventions. The reverse engineering stage mitigates this, but architect review of AI proposals is especially important for legacy codebases.

For Product Owners

Product Owners
Q If AI generates the user stories, what is my role as a PO?

Your role shifts from writing to curating, validating, and deciding. This is actually a more strategic role, not a lesser one. Specifically:

  • You define the business intent and problem statement that AI uses to generate stories
  • During Mob Elaboration, you lead the team in validating AI-generated stories for business correctness and prioritisation
  • You make the final call on acceptance criteria — AI drafts them, you own them
  • You decide when a requirement is complete enough to move to Construction (the key approval gate)
  • You own the definition of "done" — AI generates tests based on criteria you approve

What you do less of: writing stories from scratch, grooming individual items line by line, updating status manually. AI handles the generation and tracking; you handle the judgment.

Q How do "Bolts" (short AI cycles) affect my relationship with stakeholders?

Significantly — in a positive direction. Because Bolts complete in hours or days rather than weeks, you can show stakeholders working features much faster. However, it does require a change in stakeholder engagement:

  • Stakeholders need to be available for rapid feedback rather than waiting for the next sprint review (2 weeks away)
  • Requirements conversations become shorter and more frequent — Mob Elaboration sessions replace long written specs
  • The "demo sprint" cadence becomes less relevant because features ship continuously
Stakeholder Management Tip
Brief your key stakeholders on AIDLC before you start. Frame it as: "You'll see working features faster, but I'll need quick answers to questions more frequently." Set expectations that "availability for a 30-minute AI review session" replaces "review of a 20-page requirements document."
Q How does AIDLC affect the backlog in Azure DevOps?

During Mob Elaboration, AI generates work items (stories, tasks, acceptance criteria) based on the approved requirements. In our setup, these can be created directly in Azure DevOps via GitHub Copilot integration. The backlog becomes:

  • More granular — AI breaks epics into fine-grained tasks automatically
  • Better documented — each item links to the requirements and design artefacts in aidlc-docs/
  • More auditable — the audit.md file shows why each item was created and who approved it

Q What is the SM's role in Mob sessions?

The SM becomes the facilitator of Mob Elaboration and Mob Construction sessions. Key responsibilities:

  • Timebox the sessions (Mob Elaboration: target 90–120 min per significant feature)
  • Ensure all roles are present and engaged — AIDLC Mob sessions require the whole team
  • Manage the approval gates — track which phase transitions have been signed off and by whom
  • Monitor human approval latency — identify if team members are blocking AI progress by not responding to approval requests
  • Escalate impediments that prevent the team from providing AI with the context it needs

For Technical Managers

Technical Managers
Q How do I make the business case for AIDLC to senior leadership?

Frame it in terms of competitive risk and productivity ROI:

  • Competitive risk: Major industry players (AWS, Microsoft/GitHub, Google) have all formalised AI-native development methodologies. Teams that don't adopt a structured AI development practice will fall behind on velocity and cost competitiveness.
  • Productivity ROI: AWS reports 10–15x productivity gains with full AIDLC adoption. Even a conservative 2x improvement translates to significant cost savings or reallocation of engineering capacity.
  • Quality improvement: AIDLC's continuous AI-generated testing and documentation reduces defect escape rates. Less technical debt, fewer late-night incident calls.
  • Developer retention: Engineers who work with cutting-edge AI development practices report higher job satisfaction. Falling behind on tooling is a talent retention risk.

Suggested framing for leadership: "We are not adopting AIDLC as an experiment. We are adopting it because teams that don't will face a structural productivity disadvantage in 12–24 months."

Q What does AIDLC compliance and governance look like?

AIDLC has stronger built-in compliance controls than traditional SDLC because every AI action and human approval is logged:

  • audit.md: Timestamped log of every AI action, every question asked, every human approval granted. Committed to the repository.
  • Phase approval gates: No phase transition without explicit human sign-off — provides a clear decision trail.
  • Artefact versioning: Requirements, design documents, and architecture decisions are version-controlled alongside code — you always know what was agreed and when.
  • Steering rule governance: Rule files under branch protection with PR approval requirements — no unauthorised changes to AI behaviour.
For Regulated Environments
AIDLC's audit trail may actually strengthen your compliance position vs traditional SDLC. Requirements traceability (requirement → design → code → test) is auto-generated rather than manually assembled.
Q How do I evaluate whether our pilot is working?

Measure outcomes, not activity. Key metrics:

  • Cycle time: Time from work item creation to production deployment. Target: 30–50% reduction in 8 weeks.
  • Requirements rework rate: % of stories that require significant changes after Mob Elaboration. Target: under 20%.
  • Defect escape rate: Bugs found in production per feature. Track vs pre-AIDLC baseline.
  • Approval latency: Average time for humans to respond to AI approval requests. Should be under 2 hours during working hours.
  • Developer satisfaction (eNPS): Monthly pulse survey. AIDLC should improve this — if it doesn't, something is wrong with the implementation.

What to watch out for: Teams that game velocity metrics by approving AI output without proper review. Monitor defect rates alongside velocity — velocity without quality is not a win.

Security and Compliance

Architects · Tech Managers · Engineers
Q Is it safe to let AI see our codebase and requirements? What about IP and data privacy?

This is a critical governance question with practical answers:

  • GitHub Copilot (Business/Enterprise tiers): Code and prompts are not used to train the model. Data is not shared with third parties. Enterprise tier offers additional data sovereignty controls.
  • Context sent to AI: Only what you explicitly include in the prompt or what the AIDLC steering rules reference. No agent automatically uploads your entire codebase to an external server — it sends relevant context per request.
  • Sensitive data: Never include secrets, credentials, PII, or customer data in prompts or as context. This is the same rule as for code comments — treat it as if it's going into source control (because the prompts are logged in audit.md, which is).
Action Required
Before rolling out AIDLC, confirm with your organisation's data privacy officer which tier of GitHub Copilot is approved and whether there are any restrictions on code types (e.g., regulated data processing code) that should be excluded from AI-assisted development.
Q How do we prevent AI from introducing security vulnerabilities?
  • Encode security requirements in AIDLC steering rules (e.g., "always validate inputs, always use parameterised queries, never log sensitive data")
  • Run existing SAST/DAST tools on ALL AI-generated code before merge — do not skip security scanning because "AI wrote it"
  • Apply mandatory code review to security-sensitive modules (auth, payments, data access) regardless of AI generation
  • GitHub Copilot includes security vulnerability detection in its code review features — use this as a complement to, not replacement for, your existing tooling
  • Periodically audit AI-generated code patterns for systemic vulnerabilities (e.g., if AI consistently mishandles a particular pattern in your stack)
