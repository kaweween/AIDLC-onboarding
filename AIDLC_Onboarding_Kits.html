Welcome: What AIDLC Means for Your Daily Work

You are moving from a world where you write the code to a world where you guide and review AI-generated code. Your technical expertise becomes more important, not less — because you are now the quality gate between AI output and production.

Your core responsibilities in AIDLC:

  • Provide clear problem statements that give AI enough context to work from
  • Participate in Mob Construction sessions — real-time collaborative review of AI-generated code
  • Review and approve every line of AI-generated code before it merges
  • Maintain and improve AIDLC steering rules as you learn what works

Your First Week Checklist

  • Set up AIDLC rules in your primary repo

    Follow the 5-step setup below. Takes ~20 minutes. Do this before your first Mob session.

  • Read the core-workflow.md file

    Located in .github/copilot-instructions.md after setup. Understand what the AI is being told to do before you interact with it.

  • Run your first AIDLC trigger on a sandbox task

    Pick a non-critical bug or small improvement. Type "Using AI-DLC, ..." and follow the workflow through at least the Inception phase.

  • Attend your first Mob Construction session

    Even as an observer. Understand how the team interacts with AI in real-time before leading one yourself.

  • Review the semantic context document for your team

    This is the document your architect maintains that tells AI about your codebase's patterns and conventions. You need to know it to validate AI proposals.

  • Set up GitHub Copilot in Visual Studio (if not already)

    Ensure you have at least the Business tier — required for AIDLC project rules support.

Step-by-Step: Setting Up AIDLC in Your Repo

  1. Download the AIDLC workflow rules Go to github.com/awslabs/aidlc-workflows. Download the latest release zip to your Downloads folder. Extract it — you'll have an aidlc-rules/ folder with two subfolders.
  2. Set up the GitHub Copilot instruction file In your repo root, create a .github/ directory if it doesn't exist. Copy the core workflow file: 
    # On Mac/Linux
mkdir -p .github
cp ~/Downloads/aidlc-rules/aws-aidlc-rules/core-workflow.md \
   .github/copilot-instructions.md

# On Windows
mkdir .github
copy %USERPROFILE%\Downloads\aidlc-rules\aws-aidlc-rules\core-workflow.md .github\copilot-instructions.md
  3. Copy the rule details folder 
    # On Mac/Linux
mkdir -p .aidlc-rule-details
cp -R ~/Downloads/aidlc-rules/aws-aidlc-rule-details/* .aidlc-rule-details/

# On Windows
mkdir .aidlc-rule-details
xcopy %USERPROFILE%\Downloads\aidlc-rules\aws-aidlc-rule-details .aidlc-rule-details\ /E /I
  4. Add an aidlc-docs/ folder placeholder and commit 
    mkdir -p aidlc-docs
touch aidlc-docs/.gitkeep # Keeps the folder in git
git add .github/ .aidlc-rule-details/ aidlc-docs/
git commit -m "chore: add AIDLC workflow steering rules and docs folder"
git push
  5. Verify in Visual Studio Open the repo in VS. Open GitHub Copilot Chat. Confirm the rules are active by typing: "Using AI-DLC, what phase are we in?" — the AI should respond that it's ready to begin the Inception phase.

Triggering AIDLC: Prompt Examples

🟢 Good triggers — give AI clear scope and context

"Using AI-DLC, I need to add email notifications when a Jenkins build fails.
The notification should include the build name, error summary, and a link
to the build log. Our notification system uses SendGrid."

"Using AI-DLC, the customer list endpoint in the billing service is slow
when there are more than 1000 records. I want to add server-side pagination
following our REST API standards."

"Using AI-DLC, we have a new requirement: users must be able to export
their transaction history as a CSV. This is a brownfield addition to
the reporting module."

🔴 Weak triggers — too vague, AI will ask many basic questions

# Too vague — what feature? Which module? What's the expected behaviour?
"Using AI-DLC, fix the search feature"

# Too broad — AI will ask to narrow scope first
"Using AI-DLC, rebuild the user dashboard"

Code Review Standards for AI-Generated Code

Review AI-generated code the same way you would a junior engineer's code. The bar does not lower because AI wrote it.

  • Understand every line

    If you cannot explain a line in a code review, it is not ready to merge. No exceptions.

  • Run it locally

    Don't just read — trace through the logic for non-trivial functions. AI can write code that looks right but misbehaves on edge cases.

  • Check security-sensitive sections extra carefully

    Auth, data access, input validation, external API calls — spend more time on these regardless of how confident AI sounds.

  • Verify tests are meaningful

    AI generates test files automatically. Check that tests cover actual edge cases and failure modes, not just the happy path.

  • Check for AI-typical patterns

    AI sometimes introduces overly generic error handling (bare catch (Exception e)), unnecessary abstractions, or patterns from different tech stacks. Catch these in review.

  • Confirm no secrets or hardcoded credentials

    AI occasionally reproduces credential patterns from training data. Scan before merging.

Working with Jenkins in AIDLC

Jenkins remains your CI/CD engine. AIDLC's Operations phase generates infrastructure and deployment configuration. Your responsibilities:

  • Review AI-generated Jenkinsfile changes with the same rigour as application code
  • Ensure AI-proposed pipeline stages match your existing standards (build → test → SAST → deploy)
  • Do not let AI modify security scanning stages without architect review
  • AIDLC's aidlc-docs/operations/ folder will contain infrastructure decisions — review these before applying to Jenkins

What to Do When AIDLC Gets It Wrong

ProblemWhat to Do
AI generates code that doesn't match our patternsReject the PR. Flag it to your architect. Update the steering rule file to include the standard explicitly. Do not just fix it manually each time.
AI asks questions you can't answerGood — that's a knowledge gap in the team. Escalate to architect or PO before proceeding. Don't guess your way through an approval.
AI seems to "hallucinate" a library or API that doesn't existReject it immediately. AI can confidently suggest non-existent APIs. Always verify imports compile against your actual dependency tree.
AIDLC workflow gets stuck or loses contextCheck aidlc-docs/aidlc-state.md. Start a new Copilot chat session with "Using AI-DLC, continue from the last saved state in aidlc-state.md."
Team disagrees on AI's proposed architectureDo not proceed. Raise it in Mob Construction. AI should be given the consensus direction. Never approve a contested architecture decision to move things along.
The Golden Rule
AI works for you. You are responsible for everything that ships. If AI-generated code causes a production incident, the developer who approved it owns the incident response. This is not a reason to avoid AIDLC — it is a reason to review thoroughly.

Welcome: Your New Role in AIDLC

In traditional Scrum, you write user stories. In AIDLC, AI drafts user stories, and you validate, prioritise, and approve them. This shift is significant: you move from author to editor and decision-maker. The result is less time writing, more time thinking strategically.

What stays the same: you own the product vision, you define "done", and you are the final voice on what gets built and why. AI cannot replace this judgement.

What changes:

  • You lead Mob Elaboration sessions (replacing grooming) where you validate AI-generated requirements in real-time with the full team
  • You approve the Inception phase checkpoint — nothing moves to Construction without your sign-off
  • You review and approve AI-generated acceptance criteria rather than writing them
  • You engage with stakeholders on a faster cadence as Bolts complete in days rather than weeks

Your First Week Checklist

  • Understand the three AIDLC phases

    Inception, Construction, Operations. Your primary domain is Inception, but you approve phase transitions for all three.

  • Prepare your product context document

    Write a 1-page summary of your product: its users, core problems it solves, key constraints, and business goals. This becomes part of the AI's context in every Inception session.

  • Observe a Mob Elaboration session

    If your pilot team is running one, attend as an observer before leading your first one.

  • Review your Azure DevOps backlog with AIDLC in mind

    Identify which items are large enough to benefit from a full AIDLC Inception phase vs items that are small enough for direct Construction.

  • Brief your key stakeholders

    Explain that features will arrive faster, but you'll need their availability for rapid feedback rather than sprint review presentations.

  • Define your acceptance criteria standards

    AI generates acceptance criteria in formats it knows. Give it examples of what good criteria look like for your product so it matches your style.

Your Role in Mob Elaboration

Mob Elaboration is your primary new ritual. It replaces backlog grooming sessions. Here's what to expect:

📅 Mob Elaboration Session Structure (90–120 min)

  1. You present the business intent (5 min)Describe the problem or opportunity in plain language. Example: "We need users to be able to download their invoice history as a PDF. This is blocking enterprise sales."
  2. AI analyses and asks clarifying questions (15 min)AI asks about users, edge cases, constraints, integrations. You answer. So does the team. Capture decisions.
  3. AI generates a requirements draft (10 min)Requirements, user stories, and acceptance criteria appear in aidlc-docs/inception/requirements.md. Everyone reviews.
  4. Team reviews and corrects (30 min)You lead the review. What did AI miss? What's wrong? What's in scope vs out? Correct the AI through the chat interface — it updates the artefacts in real-time.
  5. You approve the Inception checkpoint (5 min)When you are satisfied with the requirements, explicitly approve them. Example: "AI-DLC, I approve the requirements in requirements.md. Please proceed to Construction." This is your formal sign-off.
Important: Don't Rush the Approval
The Inception approval gate is your most powerful control point. If requirements are wrong here, the team builds the wrong thing faster than ever before. Take the time to validate carefully — the speed gains come in Construction, not Inception.

How to Write an Effective Business Intent Statement

The quality of AI's output depends heavily on the quality of your intent statement. Here's a framework:

🟢 Strong intent statement

Context: Our enterprise customers need to reconcile their invoices with their internal accounting systems.
Problem: They currently export invoices one by one as PDFs. For customers with 100+ invoices/month, this is 2–3 hours of manual work.
Request: Add a bulk invoice export feature. Users should be able to select a date range or filter by status and download all matching invoices as a ZIP of individual PDFs or as a single consolidated PDF.
Constraints: Max 500 invoices per export. Must work for our existing customer roles (admin and finance user). Must not affect invoice viewing performance.
Definition of done: Finance users can complete a 50-invoice export in under 2 minutes.

🔴 Weak intent statement — AI will ask many basic questions

"We need an invoice export feature."

This will result in many back-and-forth clarification rounds. Prepare your intent statement before the Mob session — not in it.

Validating AI-Generated User Stories: Your Checklist

AI generates stories in a standard format. Here's what to check for each one:

  • Is the user role correct?

    AI infers user roles from context. Verify it matches your actual user types.

  • Is the business value statement meaningful?

    "So that I can manage my invoices" is weak. It should explain a real user outcome.

  • Are acceptance criteria specific and testable?

    Each criterion should be verifiable. "The export works" is not testable. "A ZIP file downloads within 10 seconds for up to 50 invoices" is.

  • Are edge cases covered?

    AI is good at happy path. Check: what happens when zero results match? When the user has no permissions? When the export is very large?

  • Is the scope bounded?

    AI sometimes generates stories that are too large. If a story will take more than 2–3 days to implement, ask AI to break it down further.

  • Does it match your prioritisation?

    AI doesn't know your business priorities — it generates stories based on completeness. You decide which ones go into the next Bolt.

Your Most Important AIDLC Habit
Be available. In AIDLC, human approval latency is the primary bottleneck. If AI is waiting for your approval and you're in back-to-back meetings for 6 hours, the team's velocity is zero. Build "AIDLC check-in" time into your calendar — 15 minutes at 10am, 2pm, and 4pm is enough to stay unblocked.
