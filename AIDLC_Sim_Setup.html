⚙️ Prerequisites

Before You Start

Confirm the following are in place for your team. AIDLC setup takes approximately 20–30 minutes per repo once prerequisites are met.

Access & Licensing

  • ✓
    GitHub Copilot Business or Enterprise — Individual tier does not support project-level instruction files needed for AIDLC. Confirm your org licence at github.com/settings/copilot.
  • ✓
    GitHub Copilot extension in Visual Studio 2022 — Install via Extensions → Manage Extensions → search "GitHub Copilot". Version 1.156+ required.
  • ✓
    Azure DevOps Contributor access on the target repo — you need push rights to the main/develop branch for the initial commit.
  • ○
    Branch policy exception (one-time) — the setup commit adds new folders and files. If your repo has a "no direct push" policy, raise a temporary exception with your tech lead, or create the setup via a PR.
  • ○
    Git installed on your machine — verify with git --version in terminal. Version 2.40+ recommended.
  • ○
    Azure Repos credential helper — ensure you can clone and push without re-entering credentials. Use git credential-manager configure.

XCode / CLI Users

💡

XCode does not natively run GitHub Copilot. For iOS/macOS projects: use Claude (add CLAUDE.md at project root instead of .github/copilot-instructions.md) or use Cursor (place rules in .cursor/rules/). The AIDLC rule content is identical — only the file location differs. CLI-only developers can use claude CLI tool and place CLAUDE.md at the project root.

Step 1 of 9
📦 Download

Download AIDLC Workflow Rules

The AIDLC rules are open-sourced by AWS Labs under the MIT-0 licence. You download them once and commit them to each repo that will use AIDLC.

ℹ️

MIT-0 Licence: No attribution required. You can modify, redistribute, and use these rules in proprietary products. Pin to a specific release tag so your team uses a known-good version.

Step 3 of 9
📝 Core Workflow

Add the Core AIDLC Steering File

The .github/copilot-instructions.md file is what GitHub Copilot reads to understand the AIDLC workflow. This single file activates AIDLC for every developer who opens this repo in Visual Studio.

Create the .github directory and copy the core workflow

PowerShell — inside billing-service/
# Create .github directory (if it doesn't exist)
$ New-Item -ItemType Directory -Force -Path ".github"
# Copy the AIDLC core-workflow.md as copilot-instructions.md
$ Copy-Item "C:\tools\aidlc-workflows\aws-aidlc-rules\core-workflow.md" ` ".github\copilot-instructions.md"
✓ Created .github/copilot-instructions.md
# Verify it's there
$ Get-Content ".github\copilot-instructions.md" | Select-Object -First 5
---
name: AI-DLC Core Workflow
description: Core workflow rules for the AI-Driven Development Lifecycle
version: 1.0
✓ File content verified

What this file does

.github/copilot-instructions.md
Markdown Preview
# AIDLC Core Workflow
## Trigger
When a user begins a prompt with "Using AI-DLC, ...", activate the AIDLC workflow. Detect whether the workspace is a greenfield or brownfield project and proceed accordingly.
## Phase Sequence
1. Workspace Detection → identify project type
2. Inception Phase → requirements and stories
3. Construction Phase → design and code
4. Operations Phase → IaC and deployment
## Human Checkpoints
Mandatory human approval required before every phase transition. Store all artefacts in aidlc-docs/ folder.
ℹ️

How GitHub Copilot reads this: GitHub Copilot Business/Enterprise automatically loads .github/copilot-instructions.md as context for every chat session in that repo. No additional configuration in VS is needed — it's convention-based. The file is version-controlled so every team member gets the same AI behaviour automatically.

The rule detail files contain phase-specific instructions. The core workflow file references them dynamically — Copilot loads the appropriate detail file at each phase transition.

Copy the rule detail files

Step 5 of 9
🧠 Semantic Context

Create Your Semantic Context Document

This is the most important team-specific file. It tells AIDLC about your codebase, standards, and architectural decisions — so AI doesn't have to guess. Your architect should own this file and keep it up to date.

Create semantic-context.md

aidlc-docs/semantic-context.md
— new file, fill in for your team
1# Semantic Context — Billing Service
2## Purpose
3This document provides context for AIDLC AI agents working in this repo.
4Update when architectural decisions change.
5
6## Project Overview
7- Service Name: billing-service
8- Tech stack: .NET 8, C#, SQL Server, Azure Service Bus
9- Auth: Azure AD, JWT tokens, role-based (Admin, Finance, Viewer)
10- API style: REST, versioned (v1, v2), OpenAPI 3.0 documented
11
12## Architectural Patterns
13- Repository pattern for all data access (never raw EF in controllers)
14- CQRS: separate Commands (write) and Queries (read) via MediatR
15- Domain events via Azure Service Bus for cross-service communication
16
17## Coding Standards
18- Always use FluentValidation for input validation — never manual if-else
19- Logging: Serilog structured logging. Always include correlation IDs.
20- No raw SQL. Entity Framework with compiled queries for performance.
21- Test framework: xUnit + FluentAssertions + NSubstitute for mocks
22
23## Security Requirements
24- All endpoints require Azure AD auth unless explicitly marked [AllowAnonymous]
25- PII must never be logged. Use [Sensitive] attribute.
26- SQL injection protection: parameterised queries only (EF handles this)
27
28## CI/CD (Jenkins)
29- Pipeline stages: Build → UnitTest → SAST(Checkmarx) → Deploy-Dev → IntTest → Deploy-Prod
30- Never remove or bypass SAST stage in Jenkinsfile changes
31- Environment configs injected via Azure Key Vault — never hardcode
✓

Best practice: Schedule a 30-minute architect session to fill in the semantic context properly. A well-written context document reduces AI clarification questions by ~60% and eliminates most standard-violation findings in code review.

🚀 Git

Commit and Push to Azure DevOps

Commit all AIDLC files and push to your Azure DevOps remote. Then configure branch policies to protect the rule files.

Stage and commit

PowerShell — billing-service/
# Check what we're about to add
$ git status
Untracked files:
.github/copilot-instructions.md
.aidlc-rule-details/
aidlc-docs/
# Stage all AIDLC files
$ git add .github/ .aidlc-rule-details/ aidlc-docs/
# Commit with a conventional commit message
$ git commit -m "chore: add AIDLC workflow rules and semantic context - Add .github/copilot-instructions.md (AIDLC core workflow) - Add .aidlc-rule-details/ (phase-specific rules) - Add aidlc-docs/ structure for generated artefacts - Add aidlc-docs/semantic-context.md with team conventions Ref: AIDLC Setup Task #4821"
[feat/add-aidlc-setup 3d7a8f1] chore: add AIDLC workflow rules and semantic context
$ git push origin feat/add-aidlc-setup
Enumerating objects: 12, done.
✓ Branch 'feat/add-aidlc-setup' set up to track remote branch

Create a PR in Azure DevOps

ℹ️

In Azure DevOps, navigate to Repos → Pull Requests → New Pull Request. Set source branch to feat/add-aidlc-setup and target to develop (or main). Add your tech lead and architect as reviewers. Use PR title: "chore: AIDLC setup — billing-service".

After merge: Set branch policy on rule files

⚠️

In Azure DevOps → Repos → Branches → main → Branch Policies, add a Required Reviewer policy for changes to .github/ and .aidlc-rule-details/ requiring approval from your architect. This prevents accidental or unauthorised changes to AI steering rules.

Troubleshooting: AIDLC not activating?

  • Copilot tier check: Ensure you're on Business or Enterprise. Individual tier doesn't load copilot-instructions.md. Check: VS → Extensions → GitHub Copilot → Account.
  • Extension version: Update GitHub Copilot extension to latest. Old versions may not support project instructions.
  • File location: The file must be exactly at .github/copilot-instructions.md from the repo root, not a subfolder.
  • Restart VS: After committing new files, close and reopen Visual Studio to refresh the Copilot context.
